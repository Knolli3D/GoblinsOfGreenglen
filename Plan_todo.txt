## Prompt 11: Updating "You Win!"-screens to match Greenglen look

You are working on “Goblins of Greenglen,” a Godot 4.6 platformer written in GDScript.

Relevant file:
- scripts/Game.gd

References:
- The attached screenshot shows the current oversized “Ouch!” death message.
- The existing main menu and completed-run menu define the desired Greenglen style.

Strategic direction:
The game currently contains six fixed levels, but it may evolve into an infinite runner. Outcome language and architecture should therefore describe a run ending rather than the player simply winning or losing.

Use these outcome names:
- Fatal damage or fall: “Run Over”
- Completing Level 6: “Run Complete”
- Replay action: “Run Again”

Replace “Ouch!”, “You Win!”, “Play Again,” and similar outcome-specific wording where appropriate.

Task:
Replace the separate death message and completed-run implementation with one reusable, outcome-driven run-result menu. Both outcomes must share the same layout, theme, controls, and lifecycle behavior while displaying appropriate content.

Required architecture:
1. Refactor the existing win menu into a neutral result component such as `run_result_menu`.
2. Introduce an explicit result type, preferably an enum such as:
   - `FAILED`
   - `COMPLETED`
3. Use one method such as `_show_run_result(result)` to populate and display both states.
4. Do not duplicate separate death and completion menu trees.
5. Keep the existing transient label only for short messages such as “Level Cleared!” Rename it if necessary so it is no longer misleadingly called `win_label`.
6. Keep all gameplay communication and progression hooks signal-based.
7. Do not redesign the broader health, scoring, quest, or level systems.

Visual design:
1. Match the existing main menu and Greenglen completed-run screen.
2. Keep the final gameplay frame visible beneath a restrained dark overlay.
3. Use the shared `ui_theme`, Cinzel fonts, Greenglen nine-patch buttons, cream text, brown outlines, and existing accent colors.
4. Use a centered Container-based layout that fits comfortably at 960x540.
5. Show:
   - outcome title;
   - final score;
   - collected coins;
   - current best completed run;
   - conditional “New Highscore!” for qualifying completed runs.
6. Use a warm muted failure accent for “Run Over” and the existing gold accent for “Run Complete.”
7. Avoid oversized labels, overlapping elements, decorative cards, or a second visual system.
8. Add two centered buttons:
   - “Run Again”
   - “Main Menu”
9. Give “Run Again” initial keyboard focus.

Outcome behavior:
- `FAILED`: shown after fatal enemy damage or a fatal fall; play death SFX and stop music.
- `COMPLETED`: shown after completing Level 6; play win SFX, process completion quests, and submit the completed run to the local highscore.
- Preserve the current policy that only completed runs update `highscore.cfg`. Do not silently make failed runs eligible. Keep `_submit_run()` as the future policy hook for endless mode.
- Both outcomes display the run’s score and coins.

Lifecycle behavior:
- Disable the finished `level_root`, but keep the result menu interactive with `PROCESS_MODE_ALWAYS`.
- “Run Again” and `R` must start a completely clean run from Level 1 through the existing centralized restart and `_load_level()` paths.
- “Main Menu” must call `_show_main_menu()` so transition tokens, music, pause state, HUD visibility, invulnerability, and level state are cleaned up consistently.
- `_load_level()` must centrally hide the result menu and restore the gameplay HUD.
- Prevent pause or stale transition UI from appearing over the result screen.
- Fatal overlap signals must not reopen or repopulate the result screen repeatedly.

Preserve:
- Score, coins, health, invulnerability, and damage tracking.
- Daily and weekly quest behavior.
- Highscore comparison and tie-breaking.
- Level-clear timing and transition-generation protection.
- Win/death SFX and music behavior.

Acceptance tests:

- Die from a fall and confirm the same result screen.
- Complete Level 6 and confirm “Run Complete.”
- Verify score, coins, best values, and conditional highscore text.
- Confirm failed runs do not update the completed-run highscore.
- Test “Run Again,” `R`, and “Main Menu” from both outcomes.
- Confirm all restart paths reset health, score, coins, damage flags, transitions, and invulnerability.

Constraints:

- No new dependencies, physics layers, or unrelated redesigns.
- Reuse the existing Greenglen theme and helpers.
- Do not commit or push unless explicitly requested.

Deliverables:

- Brief explanation of the result-state architecture and highscore policy.
- Validation results.

##Prompt 12: infinite arcade-style survival run

You are working on “Goblins of Greenglen,” a Godot 4.6 2D platformer written entirely in GDScript.

Repository:
- Root: /Users/maxi/cloude-game
- Main controller: scripts/Game.gd
- Progression autoload: scripts/Progression.gd
- Player logic: scripts/Player.gd
- Enemy logic: scripts/Enemy.gd
- Level base: scripts/Level.gd
- Levels: scenes/Level1.tscn through scenes/Level6.tscn
- Project configuration: project.godot
- Technical reference: CLAUDE.md
- Public documentation: README.md

Before changing anything:
1. Read CLAUDE.md and README.md completely.
2. Inspect all relevant scripts and scenes.
3. Check the current Git status.
4. Preserve unrelated and user-owned changes.
5. Do not commit or push unless explicitly requested.

Primary objective:
Convert the current six-level finite campaign into an arcade-style survival run that can continue indefinitely and is structurally ready for a future online leaderboard.

Do not implement networking, a server, accounts, cloud saves, HTTP requests, a launcher, or an actual online leaderboard in this task.

The game must produce one clean, immutable run-result object at the end of every legitimate run. A future leaderboard integration should be able to consume that result without redesigning the gameplay lifecycle.

Core game definition:
- A run begins at absolute Level 1.
- Reaching a goal advances to the next absolute level.
- Reaching Level 6 does not end the run.
- There is no final numbered level and no normal “You Win” termination.
- After the six current templates are exhausted, reuse them cyclically with increasing difficulty.
- A run ends only when the player reaches 0 HP.
- A voluntary restart, exit to the menu, or application shutdown abandons the run and must not finalize it as a leaderboard-eligible result.
- When a run ends through death, finalize it exactly once, update the local personal best, and expose the result through a future-proof hook or signal.
- The game must remain fully playable offline.

Required implementation areas:

==================================================
1. SEPARATE ABSOLUTE LEVEL NUMBER FROM LEVEL TEMPLATE
==================================================

The current current_level value is effectively an index into the six-entry LEVELS array. Replace or refactor this so the game distinguishes:

- absolute run level number:
  - 1, 2, 3, 4, … with no fixed maximum;
- template index:
  - 0 through LEVELS.size() - 1;
- difficulty cycle:
  - how many complete passes through the template list have occurred.

Expected mapping:

- Absolute Level 1 → Level1.tscn, cycle 0
- Absolute Level 6 → Level6.tscn, cycle 0
- Absolute Level 7 → Level1.tscn, cycle 1
- Absolute Level 12 → Level6.tscn, cycle 1
- Absolute Level 13 → Level1.tscn, cycle 2
- Absolute Level 99 → Level3.tscn, cycle 16

Use clear helpers, conceptually similar to:

- get_template_index()
- get_difficulty_cycle()
- get_absolute_level_number()

Do not use the absolute level directly as an array index.

The HUD must display the absolute level number, not the template index.

The goal at the end of every stage must load the next absolute level. Remove the special “last entry in LEVELS means victory” behavior.

==================================================
2. MAKE THE RUN END ONLY ON DEATH
==================================================

Create a clear run lifecycle:

- start new run;
- active run;
- level transition;
- run finalized;
- abandoned run;
- main menu.

Introduce explicit state or guards so the same run cannot be finalized twice.

The run must be finalized when health reaches 0, regardless of which absolute level the player reached.

The following actions must not create a finalized result:

- pressing R during an active run;
- selecting “Try Again” during an active run;
- returning to the main menu;
- closing the application;
- replacing a run with a new run.

Those actions abandon the old run and start or return cleanly.

A death followed by repeated input must not produce duplicate local records or duplicate run-finalized signals.

Remove the old final “You Win” termination. A milestone message may be shown after clearing the first six-level cycle, but it must continue into Level 7 automatically.

==================================================
3. MAKE HEALTH BELONG TO THE WHOLE RUN
==================================================

Currently _load_level() resets health to MAX_HEALTH. Change this.

Required behavior:
- A new run starts with MAX_HEALTH.
- Health persists when advancing between levels.
- Loading the next level must not fully heal the player.
- Falling without reaching 0 HP still respawns the player on the current stage.
- Reaching 0 HP ends the entire run.
- Restarting or starting a genuinely new run restores MAX_HEALTH.
- Main-menu state must not retain stale health from the abandoned run.

Do not add a complicated healing system in this task.

If a small recovery mechanic is necessary for balance, keep it explicit and simple, such as one heart after a configurable number of level clears. Prefer no automatic healing initially unless playtesting proves the endless run is unreasonably punishing.

Do not accidentally reset:
- total run score;
- total run coins;
- levels cleared;
- goblins stomped;
- run start time;
- run ID;
- run-wide damage state.

Continue resetting level-specific state, such as took_damage_this_level, on each level load.

==================================================
4. REDESIGN SCORE AS AN ARCADE RUN SCORE
==================================================

The current +1/-1 score is too small and does not represent an endless survival run well.

Create named scoring constants in one authoritative location. Use these initial values unless the existing implementation gives a strong reason to adjust them:

- Goblin stomp: +100
- Coin collected: +10
- Level cleared: +500
- No-damage level clear bonus: +250

Do not deduct score for damage or falls in the new system. Losing health is already the punishment.

Score must:
- begin at zero for a new run;
- never become negative;
- never reset between levels;
- increase immediately in the HUD;
- remain local during the active run;
- be finalized only when the run ends through death.

Continue tracking coins separately even though coins now award score.

Continue tracking goblin stomps separately.

Do not mix leaderboard score with quest counters:
- one goblin stomp still advances a stomp quest by exactly one;
- one collected coin still advances a coin quest by exactly one;
- point values must not be passed into quest progress.

Keep all scoring constants easy to rebalance later.

==================================================
5. ADD MINIMAL, FAIR DIFFICULTY SCALING
==================================================

Use the difficulty cycle or absolute level to make repeated templates gradually harder.

Keep this first version conservative and deterministic enough to test.

Recommended initial scaling:
- Increase enemy movement speed by approximately 10% per completed six-level cycle.
- Cap the multiplier at a reasonable value, such as 1.7.
- For randomized Level 6 templates, optionally increase enemy count slowly, but never beyond safe platform capacity.
- Keep enemy patrol ranges constrained to platform bounds.
- Do not change platform geometry procedurally in this task.
- Do not create impossible jumps.
- Do not increase gravity or change player jump physics.
- Do not allow enemy speed to scale forever.

Refactor Enemy.gd if necessary so enemy speed can be configured per instance or through a difficulty multiplier instead of being locked entirely in a constant.

Apply difficulty after the level scene and randomized spawns exist, so it affects both hand-placed and procedurally spawned enemies.

Keep the base behavior of the first six levels as close to the current game as possible.

==================================================
6. CREATE A FUTURE-PROOF RUN RESULT
==================================================

Create one authoritative run-finalization function, conceptually:

- _finalize_run(reason)
- finish_run(reason)
- or another clear name.

It must build a run-result Dictionary containing at least:

- schema_version
- run_id
- score
- coins
- goblins_stomped
- absolute_level_reached
- levels_cleared
- duration_ms
- death_reason
- completed
- game_version
- finalized_at

For the endless model:
- completed should normally be false because the player died rather than “won”;
- alternatively use a clearer field such as outcome = "death";
- do not falsely mark Level 6 as completion.

Generate a unique run_id when a new run starts.

Use a sufficiently collision-resistant local identifier, preferably random bytes from Godot’s Crypto API encoded as a string. Do not use only a timestamp.

Use ProjectSettings for game_version if configured, with a safe development fallback.

The run result must be built once and treated as immutable after finalization. Duplicate input must return the already finalized result or do nothing safely.

Expose the result using a clear future integration point, preferably:

signal run_finalized(result: Dictionary)

The local highscore handler can consume the same result.

Do not add placeholder HTTP calls. Do not add fake API URLs. Do not put server-specific fields throughout gameplay code.

The future online integration should eventually need only to connect to run_finalized or call one clearly documented submission hook.

==================================================
7. UPDATE LOCAL PERSONAL-BEST HANDLING
==================================================

The local highscore currently records only a completed six-level run. Change it to record the best finalized survival run, including failed runs.

A death at Level 3 with a valid score must be eligible for the local personal best.

Store at least:
- score;
- coins;
- absolute level reached;
- levels cleared;
- duration;
- run ID, if useful.

Comparison order:
1. Higher score wins.
2. If scores are equal, higher absolute level reached wins.
3. If still equal, more coins wins.
4. If still equal, shorter duration wins.

Preserve backward compatibility with existing highscore.cfg files:
- missing new fields must receive safe defaults;
- do not crash on an old save;
- do not erase an existing valid score unnecessarily.

Rename user-facing “highscore” wording to “personal best” where appropriate, while keeping file compatibility unless a migration is safely implemented.

The death screen should show:
- final score;
- coins;
- absolute level reached;
- levels cleared;
- whether it is a new personal best;
- retry instruction.

Example:

Ouch!

Score: 4,380
Coins: 28
Reached Level 9

★ New Personal Best! ★

Press R to retry

==================================================
8. DEFINE ABANDONMENT AND RETRY BEHAVIOR
==================================================

Create a clear distinction between:

- finalized run:
  - player reached 0 HP;
- abandoned run:
  - player voluntarily restarted;
  - player exited to the menu;
  - a new run replaced the old run.

Abandoned runs must:
- not update the personal best;
- not emit run_finalized;
- not become future leaderboard submissions;
- clean up all active level nodes and pending transitions;
- begin the next run with a new run ID and fresh statistics.

If R is pressed after the death screen:
- start a clean new run at absolute Level 1.

If R is pressed during an active run:
- treat it as abandonment;
- do not record the partial result;
- start a clean new run.

Preserve existing pause-menu behavior but ensure “Try Again” and “Exit to Menu” invalidate the active run appropriately.

==================================================
9. MAKE LEVEL TRANSITIONS SAFE
==================================================

The existing level-clear flow awaits a timer before loading the next level. Make it impossible for an old coroutine to load a level after:

- the player pressed R;
- the player selected Try Again;
- the player exited to the main menu;
- a new run started;
- another level load superseded the transition.

Use a transition generation/token or another robust cancellation check.

Capture the intended absolute next level before awaiting.

After the timer finishes, verify:
- the same run is still active;
- the transition is still current;
- the game is not in the main menu;
- the active level has not been replaced;
- the run has not been finalized or abandoned.

Do not permit duplicate level instances.

The current quest system includes concepts such as:

Those semantics no longer fit an endless game because normal runs end in death.

Update quest definitions and hooks so no quest becomes impossible or rewards intentionally dying quickly.

- Reach Level 12 in one run.
- Clear X levels.

- Score X points in one run.

Preserve existing quest IDs where practical for save compatibility, but update their descriptions/stat mappings safely. If an old active quest cannot be migrated cleanly, reroll it safely rather than leaving it permanently impossible.

Do not count death itself as a valuable “finish run” action.

Add progress hooks for:
- reaching an absolute level;
- current run score milestones, if used;
- level clear;
- no-damage level clear.

Avoid awarding the same level-reach or score milestone multiple times in one run.
